## 1. 決定事項（opus-context の Q1/Q2 への回答）

### Q2: core-02 は独立維持か、core-03.3 に統合か → **独立維持・問題を再定義**

- core-03.3 は「setup が完走して動くこと」= **セットアップ機構**の問題
- 再定義後の core-02 は「配布原本の**内容が完全であり続ける**こと（実働環境で生まれた開発内容が還流される）」= **リポジトリの網羅性**の問題
- setup.sh がどれだけ完璧でも、リポジトリに入っていないスキルは配れない。別の失敗モードなので統合しない
- 実行順も維持（core-02 が先行する根拠は「開発マシン⇄リポジトリの関係モデルの確定が他 core の実装着地先を決める」に変わったが、順序自体は変わらない）

### Q1: 前提変更後に残る本当の問題 → **還流経路の未定義が本丸**

opus-context の候補 A〜D への判定:

| 候補 | 判定 |
|---|---|
| A. 反映経路の明確化 | **残る。ただし本丸は逆方向**。リポジトリ→新規環境は setup.sh の仕事（core-03.3 + 明文化で足りる）。**実働環境→配布原本（還流）が未定義なのが実問題**。実働環境にのみ存在する未マージスキル群（traceability-*, lean-ctx, compact-prep）がその証拠 |
| B. worktree 残骸 | 実在するが**リポジトリ衛生の問題 → core-03.4 へ移管** |
| C. .gitignore 整合 | 同上、**core-03.4 へ移管** |
| D. core-03.3 に吸収 | **しない**（Q2 参照） |

追加で残る問題（候補に無かったもの）:
- `MM` ファイル（CLAUDE.md, skills/pr-review/SKILL.md）の突合未了。実働側の有益な変更が失われ得る
- 76PR 遅れの git clone 自体のリスク（誤 `git pull` で大量コンフリクト確定、dirty 30+ で git が実態を説明できない）→ de-git の新しい根拠

### 方向性の確定（ユーザー確認済み 2026-07-07）

- **(b) 還流ルート（実働環境 → 配布原本）の確立が必須、(a) 配備方向（配布原本 → 実働環境）は任意**。
  (a) を完璧にしても実働環境で生まれた変更は救えないため
- 還流は「検知（機械的な突合。善意に頼らない forcing function）」+「手順（検知→PR の定型ルート）」の 2 要素
- **用語変更**: 「本番」は旧前提（リポジトリが正）を含意するため廃止。
  `~/.claude/` = **実働環境**、リポジトリ = **配布原本**、戻すこと = **還流**、対象ファイル群 = **内容物**

### 旧合意の生死

| 旧合意 | 判定 |
|---|---|
| de-git | **維持**。理由を「配備方式の前提」→「無意味化した git 計器の撤去 + pull 事故防止」に差し替え |
| deploy.sh | **必須経路から格下げ**。任意の利便ツール（新 REQ-02-07）。還流方向（collect）も同じホワイトリストで実装可能 |
| ホワイトリスト定義 | **転用**。「deploy の対象」→「内容物 vs ローカルデータの区分」に意味変更。仕分け作業は無駄になっていない |
| settings.json 対象外 / ローカル専用ファイル維持 | **維持**（前提変更の影響なし） |
| agents/ は配布原本が正 | **維持**（内容物に含む）。leliel.md 削除は core-03.1 のまま。ただし deploy.sh 前提のハードブロックは解消（還流検知の「既知の削除予定物」として記録すれば足りる） |
| 旧 REQ-02-07（/finished-pr で deploy 促し） | 任意化。本番先行開発ならマージ時点で実働環境は既に最新のことが多い |
| 旧 REQ-02-08（直編集したら速やかに PR） | **昇格**。README の注意書きから、還流の中核要求（検知→PR の運用手順）へ |
